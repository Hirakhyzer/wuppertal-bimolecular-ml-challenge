## Entry 1

```text
You are helping me build a rigorous and reproducible scientific machine-learning repository for a doctoral application challenge at the University of Wuppertal.

Repository name already created 
wuppertal-bimolecular-ml-challenge

Project objective:
Build and benchmark a bi-molecular machine-learning regression model that predicts a scalar excitonic-coupling value from a pair of molecular geometries.

The provided files will be placed locally in:

data/raw/
    Coord_A.xyz
    Coord_B.xyz
    CouplingEnergies.csv
    Coord_supermol.xyz   # optional; do not assume it is required

Important rules:
1. Do not invent dataset columns, atom counts, units, output values, or results.
2. Inspect the actual files before designing assumptions around them.
3. Do not create a final report with fabricated metrics or figures.
4. All reported results must be generated by executable code.
5. Make all random processes reproducible using fixed seeds.
6. Use Python only.
7. Do not upload raw dataset files to GitHub unless explicitly permitted.
8. Add the dataset folder to .gitignore, but include data/README.md explaining expected files.
9. Create clean, documented, research-quality code, not a quick prototype.
10. Create and maintain docs/LLM_PROMPT_HISTORY.md. Add this exact prompt as the first entry. I will manually add future LLM interactions so the full prompt history is preserved.

Scientific requirements:
- Parse the concatenated XYZ files into individual molecular geometries.
- Implement or use a validated Coulomb-matrix representation.
- Build a pairwise molecular representation suitable for bi-molecular learning.
- Implement Kernel Ridge Regression as the main model using scikit-learn.
- Structure the code so Gaussian Process Regression can be added later as an optional comparison.
- Use mean absolute error as the primary evaluation metric.
- Generate a double-logarithmic learning curve:
  x-axis = number of training bi-molecular samples/pairs
  y-axis = validation MAE
- Use repeated experiments or cross-validation where computationally feasible.
- Avoid data leakage. Implement:
  (a) a standard random pair split baseline, and
  (b) a geometry-aware split where held-out molecular geometries are excluded from training.
- Clearly document the meaning and limitation of each split.
- Implement hyperparameter selection only on training/validation data.
- Use a fixed held-out test set for final model evaluation.

Implementation guidance:

1. Data inspection
- Detect the structure and columns of CouplingEnergies.csv.
- Validate whether all expected molecule-pair combinations exist.
- Report data dimensions, missing values, duplicate rows, target distribution, and molecule atom-count distribution.
- Save a concise machine-readable summary in outputs/data_summary.json.

2. Molecular representation
- Implement the Coulomb matrix:
  diagonal: 0.5 * Z_i^2.4
  off-diagonal: Z_i * Z_j / ||R_i - R_j||
- Use a robust molecular ordering strategy, such as sorting rows by norm.
- Pad representations consistently to the maximum number of atoms.
- Flatten only the appropriate upper-triangular elements or provide a documented alternative.
- Include unit tests for invariance to atom ordering where applicable.

3. Bi-molecular representation
Implement at least these pair-feature options:
- directed_concatenation: [representation_A, representation_B]
- interaction_features: [A, B, abs(A-B), A*B]

Do not assume that the target is symmetric under switching A and B.
First inspect the dataset and test whether symmetry appears plausible before adding any symmetrized representation.

4. Model
- Use Pipeline(StandardScaler(), KernelRidge(kernel="rbf")).
- Support hyperparameter search over alpha and gamma.
- Start with computationally safe configurations.
- Record all selected hyperparameters and validation MAE in structured output files.
- Ensure the full 40,000-pair dataset is not blindly used in a way that makes Kernel Ridge Regression computationally infeasible.

5. Learning curve
- Use logarithmically spaced training sizes, for example:
  100, 250, 500, 1000, 2000, 4000
  but automatically reduce the maximum size if memory or runtime becomes excessive.
- Repeat each training size across multiple random seeds where feasible.
- Save mean MAE and standard deviation/error bars.
- Create:
  outputs/figures/learning_curve_loglog.pdf
  outputs/figures/learning_curve_loglog.png
  outputs/results/learning_curve_results.csv

6. Documentation
README.md must include:
- project purpose

docs/methodology.md must explain:

7. Final deliverables
Create a one-page LaTeX report template in docs/report_template.tex with placeholders only:

First action:
Create the complete repository scaffold, requirements.txt, .gitignore, data README, source-code modules with function stubs and docstrings, configuration files, tests, and a strong README.

Then stop and provide:
```
